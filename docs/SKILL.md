# PMUI — 交互式产品原型生成器

生成专业级交互式产品原型，采用「左侧导航 + 右侧设备模型」的经典原型工具交互模式，让产品经理和开发者能快速浏览、验证产品页面流程和交互逻辑。

## 核心理念

设计灵感来自 Figma、Axure、蓝湖等专业原型工具的通用交互模式：

- **左侧导航树**：按业务模块分组的页面列表，支持折叠/展开，点击即切换页面
- **右侧设备模型**：在仿真设备外壳（手机/平板/桌面）中渲染真实页面内容
- **真实数据驱动**：使用拟真的业务数据，而非占位色块
- **完整交互逻辑**：页面间导航、表单操作、状态管理都真实可用

## 何时使用此技能

- 用户需要创建**产品原型**（任何复杂度）
- 用户需要**可点击的 demo** 来展示产品流程
- 用户需要**验证页面交互逻辑**（导航、表单、状态流转）
- 用户提到"拟真原型"、"高保真原型"、"交互原型"
- 用户在 Vibe Coding 流程中需要快速产出产品原型
- 用户需要给团队演示产品页面流程

## 文件架构

采用**四文件分离架构**，职责清晰，便于维护：

```
prototype/
├── data.js      # 数据层：所有业务数据模型（用户、商品、订单等）
├── app.js       # 状态层：全局状态管理、导航栈、业务逻辑
├── ui.js        # 视图层：所有页面的渲染函数
└── index.html   # 入口层：设备外壳、侧边栏、CSS样式、脚本加载
```

**为什么不用单文件？** 当页面超过 20 个时，单文件会变得难以维护。四文件分离让每个文件保持在合理大小（data.js ~250行, app.js ~200行, ui.js ~700行, index.html ~400行），且修改一个文件不会影响其他文件。

## 设计规范

### 1. 布局结构

```
+--------------------------------------------------+
|  +----------+  +------------------------------+   |
|  |          |  |  +------------------------+  |   |
|  |  侧边栏   |  |  |    状态栏 (模拟)        |  |   |
|  |          |  |  +------------------------+  |   |
|  |  Logo    |  |  |    导航栏 (返回/标题)    |  |   |
|  |  ------  |  |  +------------------------+  |   |
|  |  模块1 v |  |  |                        |  |   |
|  |   . 页面A |  |  |    页面内容区域         |  |   |
|  |   . 页面B |  |  |    (可滚动)             |  |   |
|  |  模块2 v |  |  |                        |  |   |
|  |   . 页面C |  |  |                        |  |   |
|  |   . 页面D |  |  +------------------------+  |   |
|  |  模块3 v |  |  |    TabBar (如适用)       |  |   |
|  |   . 页面E |  |  +------------------------+  |   |
|  |          |  |         设备外壳             |   |
|  +----------+  +------------------------------+   |
+--------------------------------------------------+
```

### 2. 设备外壳规格

| 设备类型 | 宽度 | 高度 | 圆角 | 适用场景 |
|---------|------|------|------|---------|
| iPhone | 375px | 812px | 40px | 微信小程序、移动端App |
| Android | 360px | 780px | 20px | Android应用 |
| iPad | 768px | 1024px | 16px | 平板应用 |
| Desktop | 1280px | 800px | 8px | Web应用、管理后台 |

默认使用 iPhone 外壳。外壳应包含：顶部刘海/状态栏（时间、信号、电池）、底部 Home Indicator（iPhone）、设备边框和阴影。

### 3. 侧边栏设计

侧边栏宽度固定 240px，包含：品牌区（产品名称+版本号）、导航分组（按业务模块分组，每组可折叠）、当前页面高亮（active 状态用主题色标识）。

### 4. 状态管理（app.js）

采用轻量级状态管理模式，核心包含：currentPage（当前页面）、pageStack（导航栈，支持返回）、params（页面传参）、业务状态（user/filters/selectedItems）。

关键方法：navigateTo（推入页面栈）、goBack（弹出页面栈）、switchTab（切换TabBar，清空页面栈）。

### 5. 页面渲染（ui.js）

每个页面一个渲染函数，返回 HTML 字符串。使用公共组件函数（renderStatusBar/renderNavBar/renderTabBar/showToast）避免重复。事件绑定使用 onclick 内联方式。

**重要**：HTML 属性中传递字符串参数时，使用 `&#39;` 代替 `\'`（避免 JS 语法错误）。

### 6. 数据模型（data.js）

使用真实、有意义的业务数据。数据要真实可信，反映实际业务场景。数量适中（列表 6-20 条），字段完整，关联关系正确。

### 7. 平台规范

不同平台和设备有独特的设计规范，需严格遵循。

- **iOS/iPhone**: 详见 [平台设计规范 - iOS](references/platform-specs.md#ios-iphone-平台规范)
- **Android**: 详见 [平台设计规范 - Android](references/platform-specs.md#android-平台规范)
- **iPad 平板**: 详见 [平台设计规范 - iPad](references/platform-specs.md#ipad-平板规范)
- **Desktop/Web**: 详见 [平台设计规范 - Desktop](references/platform-specs.md#desktopweb-平台规范)

核心原则：
- iOS: 扁平化、圆润、强调手势操作
- Android: Material Design、阴影、波纹反馈
- iPad: 信息密度高、分栏布局、充分利用空间
- Desktop: 响应式、导航清晰、内容最大宽度限制

## 工作流程

1. **理解需求**：确认产品类型、核心页面、业务数据、设备类型
2. **规划页面结构**：按业务模块分组列出所有页面
3. **生成代码**：按 data.js → app.js → ui.js → index.html 顺序生成
4. **验证修复**：语法检查（`node -c`）、启动预览、浏览器验证、修复 bug

## 常见陷阱和解决方案

### 1. HTML 属性中的引号嵌套

在单引号 JS 字符串中，`\'` 会导致语法错误。使用 HTML 实体 `&#39;` 代替：

```javascript
// 错误
'<div onclick="AppState.navigateTo(\'page\')">点击</div>'
// 正确
'<div onclick="AppState.navigateTo(&#39;page&#39;)">点击</div>'
```

### 2. ui.js 文件过大

当页面超过 30 个时，使用 Python 脚本批量生成重复性高的页面，将公共组件抽取为辅助函数。

### 3. 页面间参数传递

```javascript
// 发起导航
AppState.navigateTo('orderDetail', { orderId: 'ORD001' });
// 目标页面读取
const order = AppData.orders.find(o => o.id === AppState.params.orderId);
```

### 4. 导航栈管理

- `navigateTo`：用于二级页面（有返回按钮），推入 pageStack
- `switchTab`：用于 TabBar 页面，清空 pageStack
- `goBack`：弹出 pageStack 最后一项，回到上一页
